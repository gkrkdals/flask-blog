답변

안녕하세요! 20년 차 시니어 엔지니어 관점에서 프로젝트를 살펴보았습니다. 작성해주신 코드는 Flask 기반의 경량화된 구조를 잘 띄고 있지만, 규모가 커질 경우 유지보수와 확장을 어렵게 만드는 코드 악취(Code Smell)들이 곳곳에 숨어 있습니다.

요청하신 5가지 기준(Refactoring 분류)에 따라 현재 코드의 문제점 5가지를 리뷰하여 정리해 드립니다.

---

### 1. 비대해진 코드 (Bloaters) : 거대한 메서드와 기본 타입 집착
* **위치**: post_controller.py 내의 `home`, `post_detail`, `create_post` 함수들
* **설명 (Long Method & Primitive Obsession)**: 
컨트롤러 내에서 비즈니스 로직을 처리해야 할 함수들이 HTML 문자열을 하드코딩하여 통째로 반환하고 있습니다. 원시 타입(String)에 UI 로직을 과도하게 우겨넣다 보니, 기능이 조금만 추가되어도 함수가 금방 비대해질 것입니다.
* **개선 방향**: Jinja2와 같은 템플릿 엔진을 도입하고 `render_template`을 활용해 컨트롤러와 뷰(HTML) 구조를 물리적으로 분리해야 합니다.

### 2. 객체지향 남용 (Object-Orientation Abusers) / 부족 : 클래스 및 모델 부재
* **위치**: login_controller.py 모듈 상단 (`TEST_ID`, `TEST_PWD`) 및 게시물 관리 로직
* **설명 (Primitive Obsession)**: 
Python이라는 객체지향 언어를 사용하면서도 User나 Post에 대한 도메인 모델(객체)이 전혀 정의되지 않았습니다. 계정 정보를 단순히 전역 원시 타입(Global Primitive Variable)으로 관리하고 있으며, 데이터(사용자 정보)와 행동(인증 로직)이 응집되지 못하고 절차지향적으로 나열되어 있습니다. 
* **개선 방향**: 인증 로직을 전담하거나 계정 정보를 다루는 `User` 객체, 그리고 게시글 데이터를 다루는 `Post` 객체를 만들어 도메인의 책임을 부여해야 합니다.

### 3. 변경 방해자 (Change Preventers) : 산탄총 수술 (Shotgun Surgery)
* **위치**: post_controller.py 내 `create_post` 함수의 인증 처리 로직 (`if not session.get("logged_in"):`)
* **설명**: 
현재 `create_post` 내에 인증 처리 로직이 강하게 하드코딩 되어 있습니다. 향후 '글 수정', '글 삭제', '내 정보 보기' 등 인증이 필요한 API가 10개로 늘어나면 이 로직이 10군데에 복사&붙여넣기 되어야 합니다. 나중에 세션 키 텍스트가 `"logged_in"`에서 다른 값으로 변경되거나 인증 방식(JWT 등)이 바뀌면 앱 전체 파일들을 열어 산발적으로 흩어진 코드를 모두 뜯어고쳐야 합니다. (파편화 현상)
* **개선 방향**: `@login_required`와 같은 데코레이터를 만들거나 미들웨어를 도입해 인증 로직을 한 곳으로 모아야 합니다.

### 4. 불필요한 코드 (Dispensables) : 중복 코드와 잘못된 위치의 임포트
* **위치**: post_controller.py 및 test_auth.py
* **설명 (Duplicate Code / Dead Code)**: 
  * 컨트롤러 반환 문자열들에 링크 태그 (`<a href='/'>`, `<h2>` 등) 같은 레이아웃 뷰 스크립트들이 불필요하게 흩어져서 중복(Duplicate code)되고 있습니다.
  * `create_post` 함수 내부를 보면 지역적으로 `from flask import session`를 수행하고 있습니다. 상단에서 임포트해도 무방한 모듈을 함수 내부에 위치시켜 읽는 흐름을 방해합니다.
* **개선 방향**: 공통 레이아웃을 포함하는 `base.html` 템플릿을 만들어 중복을 제거해야 하며, 패키지 `import`문들은 린트(Lint) 규칙에 맞춰 모듈 최상단으로 정리하는 것이 좋습니다.

### 5. 결합도 유발자 (Couplers) : 부적절한 친밀도 (Inappropriate Intimacy / Feature Envy)
* **위치**: post_controller.py의 `create_post` 함수 내부 `session` 참조
* **설명**: 
글 작성을 담당하는 `post_controller` 모듈이, 로그인 처리를 관리하는 `login_controller`나 서버 인증 세션(Session)의 내부 구체적인 상태 키맵(`logged_in`)을 지나치게 꿰뚫어 보고 있습니다. 도메인 간의 결합도가 너무 높아 "부적절하게 친밀한" 상태입니다.
* **개선 방향**: 게시물을 작성하는 컨트롤러는 "현재 사용자가 인가된 상태인가?"라는 메시지만 던지도록 하고 구체적인 세션 검사는 철저히 캡슐화해야 합니다.

당신이 식별한 5가지 문제를 리팩토링하기 위해 여러 리팩토링 전략을 제안해주세요.
----------------------------------------------------------------------------
답변

앞서 식별된 5가지 코드 악취(Code Smell)를 해결하기 위해 20년 차 시니어 엔지니어로서 구체적인 방향과 함께 적용할 수 있는 리팩토링 전략 4가지를 제안해 드립니다.

### 1. 뷰(View) 템플릿 추출 (Extract Template)
**적용 대상:** 비대해진 코드(Bloaters), 불필요한 코드(Dispensables - HTML 중복)
* **전략:** 컨트롤러 내부에 문자열로 하드코딩된 HTML 코드 덩어리들을 Jinja2 템플릿 엔진을 사용하여 분리합니다.
* **구현 방법:**
    1. 프로젝트 루트에 `templates` 폴더를 생성합니다.
    2. 중복되는 레이아웃(헤더, 내비게이션, 푸터 등)을 묶어 `base.html`로 구성합니다. (Extract Superclass와 유사한 템플릿 상속 개념 적용)
    3. `home.html`, `post_detail.html`, `create_post.html` 등의 개별 뷰 파일을 만듭니다.
    4. 컨트롤러에서는 로직만 처리하고 `return render_template('home.html', posts=posts)` 형태로 데이터만 넘겨주도록 변경합니다.

### 2. 데코레이터 패턴 도입 (Introduce Decorator Pattern)
**적용 대상:** 변경 방해자(Change Preventers - 산탄총 수술), 결합도 유발자(Couplers - 부적절한 친밀도)
* **전략:** post_controller.py가 로그인 세션 상태(`logged_in`)를 직접 검사하는 로직을 분리해 단일 책임 원칙(SRP)을 지키게 합니다.
* **구현 방법:**
    1. 인증을 담당하는 유틸리티 폴더나 모듈(예: `utils/auth.py`)을 생성합니다.
    2. 파이썬의 `functools.wraps`를 활용하여 `@login_required` 데코레이터를 구현합니다.
    3. 이를 활용하면 post_controller.py 내부에서는 `session`을 알 필요 없이, 함수 위에 `@login_required`만 붙임으로써 세션 결합도를 끊어낼 수 있고, 인증 로직이 변경되더라도 데코레이터 한 곳만 수정하면 됩니다.

### 3. 원시 타입을 객체로 전환 (Replace Data Value with Object)
**적용 대상:** 객체지향 남용/부족(OO Abusers - 기본 타입 집착)
* **전략:** 현재 login_controller.py 상단에 하드코딩된 원시 타입 전역 변수(`TEST_ID`, `TEST_PWD`)와 post_controller.py의 더미 게시물 데이터를 객체 모델로 승격시킵니다.
* **구현 방법:**
    1. `models` 디렉터리를 만들고 `user.py`, `post.py`를 생성합니다.
    2. 데이터베이스를 연결하기 전이더라도 `User` 클래스와 `Post` 클래스를 정의하고, 로그인 검증 로직(`User.authenticate(user_id, password)`)이나 게시글 조회 로직(`Post.get_all()`) 등을 해당 클래스의 책임(메서드)으로 위임합니다.
    3. 컨트롤러는 이 모델 객체의 메서드를 호출하기만 하도록 역할을 축소합니다.

### 4. 선언문 이동을 통한 모듈 정리 (Move Statements / Organize Imports)
**적용 대상:** 불필요한 코드(Dispensables - 잘못된 위치의 임포트)
* **전략:** 함수 내부에 숨어 있는 `import` 구문을 파일의 최상단으로 끌어올립니다.
* **구현 방법:**
    1. post_controller.py의 `create_post()` 내부 등에 지역적으로 선언된 `from flask import session`을 찾아냅니다.
    2. 파이썬 PEP-8 스타일 가이드에 따라 파일의 최상단으로 옮겨 모듈의 의존성을 한눈에 파악할 수 있도록 리팩토링합니다. (데코레이터 패턴을 도입하면 이 부분은 자연스럽게 다른 파일로 분리되어 사라지게 됩니다.)

답변

뷰 템플릿 추출(Extract Template) 리팩토링을 완료했습니다! 다음과 같은 작업을 수행했습니다.

1. **템플릿 디렉토리 및 파일 생성**
   - 뷰 템플릿들을 보관할 templates 폴더를 생성했습니다.
   - HTML 구조의 중복을 제거할 공통 레이아웃 파일인 base.html을 작성했습니다.
   - 각 페이지의 세부 정보를 담당하는 템플릿 파일(home.html, post_detail.html, create_post.html)을 생성하고, Jinja2 상속(`{% extends "base.html" %}`)을 적용하여 HTML을 정리했습니다.
2. **컨트롤러 리팩토링 (post_controller.py)**
   - 하드코딩되어 코드를 비대하게 만들던 긴 HTML 문자열을 모두 제거했습니다.
   - 대신 `render_template`을 사용하여 위에서 작성한 HTML 템플릿 파일들을 반환하도록 변경했습니다.
   - `post_detail(post_id)`의 경우 템플릿 엔진 렌더링 시 매개변수로 `post_id`를 잘 전달하도록 수정했습니다. 

이 작업을 통해 화면 렌더링에 대한 책임을 뷰 영역으로 분리함으로써 **비대해진 코드(Bloaters)**와 **중복된 UI 레이아웃의 불필요한 코드(Dispensables)** 악취를 성공적으로 제거했습니다.
